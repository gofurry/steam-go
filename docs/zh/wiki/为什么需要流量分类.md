# 为什么需要流量分类

流量分类存在的原因是：并不是所有 Steam 相关 HTTP 请求都是同一种请求。

## 问题

简单 SDK 可能把所有请求都当成一类：

```text
one client
one retry policy
one proxy
one rate limit
one cookie jar
```

初期这样很简单，但 SDK 成长后会受到限制。

## 不同流量有不同需求

| 流量 | 需求 |
|---|---|
| 官方 Web API | keys、typed JSON、retry、rate limit |
| 公开 Store 页面 | 浏览器请求头、Referer、cache、block detection |
| OpenID 验证 | callback 安全、可能需要代理、严格 state 校验 |
| 未来类 session 流程 | cookies、sticky proxy、低并发 |

## SDK 设计

`steam-go` 使用显式流量分类：

```go
steam.TrafficClassOfficialAPI
steam.TrafficClassPublicStorePage
```

然后通过 `TrafficPolicy` 按类别配置行为。

## 好处

- 避免把 API 请求和类浏览器请求行为混在一起。
- 保持业务 API 方法清晰。
- 让未来 Store 页面支持更容易。
- 可以按流量类型设置更安全默认值。
- 让 proxy、cookie、cache 行为显式化。

## 示例

```go
client, err := steam.NewClient(
    steam.WithTrafficPolicy(
        steam.TrafficClassPublicStorePage,
        steam.TrafficPolicy{
            Cache: &steam.TrafficCachePolicy{
                TTL: time.Minute,
            },
        },
    ),
)
```

## 设计原则

先识别流量类型，再应用合适策略。
