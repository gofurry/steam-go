# 限流策略

`steam-go` 使用相对保守的请求控制策略，因为 Steam Web API 同时存在官方限制和真实场景中的网络边界。

## 为什么需要限流

Steam Web API Terms of Use 明确限制应用每天最多 100,000 次调用。

除了每日限制，生产环境也不应该发起过于激进的突发请求。外部服务可能返回 `429`，也可能临时降级，或者对看起来异常的流量进行限制。

## SDK 策略

`steam-go` 没有把限流写死，而是提供多层能力：

- `WithSafeDefaults()`
- `WithRateLimit(...)`
- `WithRateLimiter(...)`
- `WithRetry(...)`
- `WithRetryBackoff(...)`
- `WithRetryRespectRetryAfter(...)`
- `WithHealthCheckedAPIKeys(...)`
- `TrafficRateLimiterPolicy`
- `HostControl`
- `SessionControl`

## 安全默认值

普通外部请求可以先使用：

```go
client, err := steam.NewClient(
    steam.WithSafeDefaults(),
)
```

这个配置故意偏保守，后续可以按业务场景调整。

## 自定义限流器

较重的业务可以显式配置：

```go
client, err := steam.NewClient(
    steam.WithRetry(2),
    steam.WithRateLimiter(rate.Limit(5), 5),
    steam.WithRetryRespectRetryAfter(true),
)
```

## 按流量类型限流

公开 Store 页面类流量可以单独降低请求速度：

```go
client, err := steam.NewClient(
    steam.WithTrafficPolicy(
        steam.TrafficClassPublicStorePage,
        steam.TrafficPolicy{
            RateLimiter: &steam.TrafficRateLimiterPolicy{
                Limit: 2,
                Burst: 2,
            },
        },
    ),
)
```

## 实用建议

- 不要用无限 goroutine 直接打 Steam API。
- 遇到 `429` 应该降速，而不是继续重试轰炸。
- 重复读取优先使用缓存。
- 公开 Store 页面访问应使用更低并发。
- official API 流量和 Store 页面类流量应拆分 traffic class 和预算。
- 多 key 轮换是为了弹性，不是为了绕过限制。
- 不要把非官方的每秒固定限流值写成事实。
- 限流应尽量放在 SDK 边界，而不是散落在业务逻辑深处。

## Storefront AppDetails 经验实验

2026-06-07 的一次内部实验使用如下配置测试 Storefront `appdetails`：

- 每轮 `360` 个逻辑请求。
- `10` workers。
- 无本地 Store interval。
- 无本地 cooldown。
- 无 retry。
- 无 proxy。

两轮实验间隔约 5 分钟。两轮都成功了 `228` 个请求，随后剩余 `132` 个请求失败。

| Run ID | OK | Failed | HTTP 429 | HTTP 403 | Transport |
|---|---:|---:|---:|---:|---:|
| `20260607-183228` | 228 | 132 | 31 | 74 | 27 |
| `20260607-184100` | 228 | 132 | 34 | 98 | 0 |

结论解释：

- `429` 大约从 220-230 次 Store appdetails 请求附近开始出现。
- 继续请求后会快速转为 `403` / block-detected。
- 等待约 5 分钟后，复测可以再次到达相近的请求数量边界。
- 这支持 Store 约 `150-250` requests / 5 minutes / egress identity 的保守预算判断。

生产访问 Storefront appdetails 时，可以先按约 `1 request / 2 seconds`、`burst=1` 设计，再基于缓存、队列和实际 `429` / `403` 观察逐步调优。

详细报告：[docs/zh/experiments/store-rate-limit-20260607.md](../experiments/store-rate-limit-20260607.md)。

## 推荐生产策略

| 场景 | 建议策略 |
|---|---|
| 小工具 / CLI | `WithSafeDefaults()` |
| 后端服务 | `WithRetry(2)` + 显式限流器 |
| 公开 Store 页面访问 | 独立 Store traffic class + 低 RPS + 缓存 + 拦截检测 |
| 大批量任务 | 队列 + worker pool + 全局限流 |
| 多 key 场景 | 健康检查 key provider + retry |
